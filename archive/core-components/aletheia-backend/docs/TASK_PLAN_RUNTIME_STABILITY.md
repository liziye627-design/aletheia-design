# Task Plan: 后端先跑稳（Runtime Stability First）

## 1. 目标（当前阶段）
在**不优先处理安全/鉴权改造**的前提下，让 `aletheia-backend` 能稳定启动、核心接口可用、运行中不出现明显错误（崩溃/大面积 5xx/卡死）。

## 2. 范围
- 启动链路：配置加载、数据库初始化、缓存可降级、后台任务启动/关闭。
- 核心链路：`/health`、`/api/v1/investigations/*`、`/api/v1/reports/*`、`/api/v1/rss/*`。
- 运行稳定性：超时、并发、异常处理、日志可观测性。
- 非目标（后续再做）：鉴权、安全加固、SSRF、路径注入等安全问题。

## 3. 执行计划

### Phase A: 先保证“能启动、能跑通”
- [ ] A1. 梳理并冻结最小可运行配置（`.env.example` + 启动脚本）。
- [ ] A2. 启动时依赖降级策略统一：Redis/MediaCrawler/外部 LLM 不可用时不阻塞主服务。
- [ ] A3. 修复启动过程中的显性异常（导入错误、配置错误、路径错误）。
- [ ] A4. 建立一键自检命令（健康检查 + 核心 API 冒烟）。

### Phase B: 先消除明显运行时错误
- [ ] B1. 修复最容易触发 500 的接口分支（参数边界、空值处理、异常分层）。
- [ ] B2. 统一错误返回（避免堆栈式噪声），保证可诊断且不影响链路。
- [ ] B3. 修复后台任务生命周期问题（重复启动、未关闭、超时僵死）。
- [ ] B4. 修复 SQLite 并发/阻塞造成的明显卡顿（优先最小改动方案）。

### Phase C: 稳定性基线
- [ ] C1. 加入运行时保护：关键超时、有限重试、降级分支。
- [ ] C2. 控制高消耗任务并发，避免运行中资源打爆。
- [ ] C3. 完成最小回归测试集（重点接口 + 错误路径）。
- [ ] C4. 形成“已知限制与规避方式”文档。

## 4. 验收标准（Done Definition）
- [ ] 服务可稳定启动 3 次以上，无阻塞性异常。
- [ ] 核心健康检查：`/health`、`/health/ready`、`/health/live` 正常。
- [ ] 核心接口冒烟通过，连续请求下无明显 5xx 峰值。
- [ ] 长时间运行（>=30 分钟）无明显内存泄漏/任务堆积。
- [ ] 日志可定位问题（错误有上下文，非噪声刷屏）。

## 5. 回归与发布策略
- 小步提交：每个 Phase 独立提交与验证。
- 先本地冒烟，再进行短压测（低并发）。
- 出现回归时优先回滚最近稳定点。

## 6. 风险与备注
- 当前阶段为了先跑稳，安全相关问题暂记在后续修复清单。
- 若外部依赖波动（LLM/爬虫站点）导致失败，应优先确保降级可用而不是硬失败。

## 7. 后续（第二阶段）
- 安全与鉴权整改（Auth、SSRF、防路径滥用、错误信息泄露）。
- 完整限流与配额治理。
- 性能与成本优化。
